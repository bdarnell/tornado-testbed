popular Python packages that depend on Tornado, each isolated in its own

version from source (or, where the source build needs a JS toolchain or

## How packages were selected

Candidates were filtered to those with a direct `tornado` runtime dependency
(appearing explicitly in their `install_requires` / `pyproject.toml` /
`requirements.txt`) and then ranked by a combination of GitHub stars and
PyPI download volume. The top ten include:

- **Jupyter stack** (jupyter_server, notebook, jupyterhub, ipykernel,
  voila) — Tornado is the HTTP engine.
- **Viz stack** (bokeh, panel) — Bokeh server is Tornado-based; Panel builds
  on Bokeh.
- **Data / ML** (streamlit, distributed, flower) — Streamlit uses Tornado
  for its web server; distributed uses it for comms; Flower is a Tornado
  app for Celery monitoring.

## Skipping policy, as exercised

The task brief said: skip suites that are "exceptionally hard to run." In
practice I pointed each `test_cmd` at a focused module or two that:

- Exercises the package's Tornado integration points where possible
  (e.g. `tests/unit/bokeh/server/test_tornado__server.py`).
- Avoids heavy prerequisites that aren't about Tornado: JS/node toolchains,
  selenium/playwright browsers, live databases, or external network fetches
  for CSS/JS assets.
- Avoids known-broken combinations unrelated to Tornado (see the
  `jupyter_server` notes in `packages.json`: its full
  `test_serverapp.py` trips a `jupyter_events>=0.12` schema-version
  incompatibility that is not a Tornado issue).

Two packages could not be built from source in this sandbox and are
installed from their published wheel instead:

- **panel**: source build runs a full `npm` + Bokeh-asset bundling step
  (`hatch_build.py` -> `bundle_resources()`), which failed even with
  `nodejs` available.
- **voila**: source build tries to fetch bundled CSS over HTTP and got
  `403 Forbidden` in the sandbox.

In both cases I still run the package's own test files from the cloned
checkout, just against the pre-bundled wheel installed into the venv.
The manifest's `install_method: "pypi"` + `pypi_spec` fields drive this.

## How to re-use the harness

```bash
./scripts/setup.sh                         # clone all packages
./scripts/run_all.sh                       # run everything
./scripts/run_one.sh <name-or-index>       # run one
./scripts/summarize.sh                     # refresh results/summary.txt

# Point it at any Tornado build:
TORNADO_SPEC="tornado==6.5.0"          ./scripts/run_all.sh
TORNADO_SPEC="tornado==7.0.0.dev1"     ./scripts/run_all.sh
TORNADO_SPEC="/abs/path/to/wheel.whl"  ./scripts/run_one.sh bokeh
```

Each call to `run_one.sh` builds a fresh per-package venv, so there is no
state carried between packages.

## Gotchas worth knowing if you extend the harness

These bit me during initial test runs — if you add a new package, expect to
hit one or more of them:

- **`uv venv` has no `pip` by default.** Any test command that shells out
  to `pip` must use `uv pip` instead.
- **setuptools_scm / hatch-vcs packages** (bokeh, distributed) want full
  tags. `setup.sh` uses `--depth 1 --no-single-branch` + `fetch --tags` so
  `git describe` works.
- **Project-level `pytest.ini` / `pyproject.toml`** often carries
  `--cov=...`, `timeout`, `asyncio_mode` config that fails when the
  matching plugin isn't installed. Either install the plugin
  (`uv pip install pytest-timeout pytest-asyncio`) or override per-run
  with `-o addopts=`.
- **Strict mode + filterwarnings=error** (common in the Jupyter stack)
  will turn benign downstream warnings into failures. Pick a smaller
  test target rather than fighting the warning filter globally.
- **Protobuf codegen** for streamlit is a hard prereq (any `import
  streamlit` fails without it); the manifest's `setup_extra` hook
  runs `protoc` for that one package.
