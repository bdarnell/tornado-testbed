# Downstream Tornado Test-Suite Report

Generated by running `scripts/run_all.sh` against pinned refs of the 10 most
popular Python packages that depend on Tornado, each isolated in its own
`uv`-managed Python 3.11 virtualenv. Every run installs the package's pinned
version (from source, or the PyPI wheel where the source build needs a JS
toolchain or network assets), then forcibly upgrades `tornado` to the
`TORNADO_SPEC` — here the pinned stable **6.5.5**.

This iteration focused on **expanding the test runs to exercise more of each
upstream suite using real test servers** (rather than the previous tiny,
mostly-import-level subsets), with a time budget of up to ~5–10 min per
project. The Jupyter stack and the HTTP/WebSocket layers of Tornado were the
main beneficiaries.

### What didn't move, and why

- **`tornado.auth` stayed at 18%.** Flower is the only package in the set that
  imports `tornado.auth`, and its committed test suite (already run in full —
  162 tests) only exercises the `authenticate`/`validate_auth_option` helpers
  and HTTP Basic auth. It has **no tests that drive the OAuth2 login handlers**
  (`GoogleOAuth2Mixin.get_authenticated_user`, `OAuth2Mixin.authorize_redirect`,
  `oauth2_request`), which is where the bulk of `auth.py` lives. Raising this
  number would require writing new OAuth-flow tests, which is explicitly out of
  scope for this task, so `auth.py` coverage is unchanged.

## Per-package notes on the expansion

- **jupyter_server (6% → 42%).** Switched from four pure-function modules to
  the entire `tests/` tree (~970 tests) driven by the `pytest-jupyter`
  fixtures, which boot a real `ServerApp` on a real port and exercise it with
  `jp_fetch` / `jp_ws_fetch`. The `jupyter_events>=0.12` schema-version
  incompat that previously blocked these tests is sidestepped with a targeted
  `-W ignore::...JupyterEventsVersionWarning` (jupyter_server v2.14.2 ships
  integer schema versions; newer jupyter_events warns and the project's
  `filterwarnings=error` turns that into a failure — unrelated to Tornado).
  Three of its own order-dependent / subprocess-launch tests are deselected.

- **jupyterhub (18% → 42%).** Switched from logic-only modules to the live
  `test_api` / `test_services_auth` / `test_metrics` / `test_dummyauth`
  suites, which boot a real MockHub Tornado application plus a
  `configurable-http-proxy` node process. **Prerequisite:** the
  `configurable-http-proxy` binary must be installable via `npm` (the test_cmd
  installs it globally if missing). `test_pages`/`test_named_servers` were left
  out because they spawn real single-user notebook servers and overrun the time
  budget.

- **ipykernel (3% → 7%).** Now runs the real-kernel suite (~140 tests), which
  starts IPython kernels over ZeroMQ on Tornado's asyncio loop —
  `platform.asyncio` 41%, ioloop 58%, queues 51%, concurrent 38%. The editable
  install's auto-generated kernelspec points at uv's (now-cleaned) build-time
  Python, so the test_cmd reinstalls a correct one with `ipykernel install
  --sys-prefix` first. The overall % stays low because ipykernel touches none
  of the web/HTTP/WebSocket layers. `test_message_spec` (nose-style module
  `setup()` that pytest 9 no longer calls) and a few debugger/IPython
  version-specific tests are deselected.

- **voila — evaluated, kept at baseline (20%).** Voila's `tests/app` and
  `tests/server` integration suites *do* drive real Tornado servers and execute
  notebooks over a real kernel WebSocket, and when they complete they lift
  coverage to ~30% (websocket 63%). They were **deliberately not adopted as the
  default** because, under the harness's always-fresh venv, the cold-start
  kernel-WebSocket tests deadlock intermittently (~50% of runs) in a ZeroMQ poll
  that swallows both `pytest-timeout`'s SIGALRM and a plain SIGTERM — i.e. they
  hang the run until SIGKILL. A coin-flip package is worse than a smaller
  reliable one, so voila stays on `utils_test.py`. `packages.json` documents how
  to opt into the fuller suite manually.

- **notebook, flower, streamlit, bokeh, distributed, panel — unchanged tests.**
  notebook has a single test file (already run in full; its 23% now reflects the
  reporting fix). flower already runs its whole suite. The rest were left as-is;
  this iteration's focus was the Jupyter stack and `tornado.auth`.

## Reporting fix (`--cov-config=/dev/null`)

The previously committed `coverage_html/{jupyter_server,notebook,ipykernel}`
reports showed **0%** — a path bug, not zero coverage. Those three projects set
`relative_files = true` in their own coverage config, so pytest-cov stored
source paths like `.venv/lib/.../tornado/web.py`; `coverage html` then couldn't
line them up and marked everything missing. Adding `--cov-config=/dev/null` to
those three `test_cmd`s makes pytest-cov ignore the project config and record
absolute paths (as the other seven packages already did), so their reports now
render correctly.

## Panel robustness fix (`-p no:unraisableexception`)

Panel's `filterwarnings=["error"]` promotes a GC-timing
`ResourceWarning: unclosed socket` (leaked by its Bokeh/Tornado server
fixtures) into a hard error during pytest's end-of-session unraisable-exception
sweep — all 40 tests pass, only teardown fails. Disabling that sweep with
`-p no:unraisableexception` restores a reliable PASS without masking anything
Tornado-related.

## How to re-use the harness

```bash
./scripts/setup.sh                         # clone all packages
./scripts/run_all.sh                       # run everything
./scripts/run_one.sh <name-or-index>       # run one
./scripts/summarize.sh                     # refresh results/summary.txt
./scripts/gen_reports.sh                   # rebuild coverage_html/

# Point it at any Tornado build:
TORNADO_SPEC="tornado==6.5.0"          ./scripts/run_all.sh
TORNADO_SPEC="tornado==7.0.0.dev1"     ./scripts/run_all.sh
TORNADO_SPEC="/abs/path/to/wheel.whl"  ./scripts/run_one.sh bokeh
```

Each call to `run_one.sh` builds a fresh per-package venv, so there is no state
carried between packages. The live-server suites need a couple of system
prerequisites: **node/npm** (for jupyterhub's `configurable-http-proxy`) must be
on PATH.

## Gotchas worth knowing if you extend the harness

- **`uv venv` has no `pip`** — test commands that shell out to `pip` must use
  `uv pip`.
- **setuptools_scm / hatch-vcs packages** (bokeh, distributed) want full tags;
  `setup.sh` fetches them.
- **Project-level `filterwarnings=error`** (the whole Jupyter stack) turns
  benign downstream warnings into failures. Prefer a *targeted*
  `-W ignore::Specific.Warning` over fighting the global filter or shrinking the
  test target.
- **`relative_files=true`** in a project's coverage config yields relative paths
  that break standalone `coverage html`; pass `--cov-config=/dev/null` to force
  absolute paths.
- **Real-kernel (ZeroMQ) tests can deadlock** in a way that ignores SIGALRM and
  SIGTERM; wrap them in `timeout -s KILL` if you must run them unattended, and
  expect intermittency.
- **Protobuf codegen** for streamlit is a hard prereq; the manifest's
  `setup_extra` hook runs `protoc` for that one package.
